## 1. 応援決済（チア）の決済セッション作成

**テストの目的・背景:**  
ファンがアーティストに応援を送る際、カード・PayPay・Apple Pay など複数の決済手段が正しく分岐し、オーガナイザー（主催者）の Stripe Connect 口座設定状態によって処理が変わることを保証する。「MoR（決済の代理人）はオーガナイザー」というプラットフォームの核心設計が守られているかを検証する。

**網羅している観点（境界値・状態）:**
- 決済手段: カード / Apple Pay / Google Pay / Link / PayPay の5手段
- オーガナイザーの口座状態: 設定済み・未設定・審査中（Capability不足）
- PayPay の特殊ルール: Connect の `on_behalf_of` 非対応のため「プラットフォームがMoR」になる仕様
- 必須フィールド欠損、ゼロ金額などのバリデーション

**主なテストケースと期待される結果:**
- [正常系] カード決済でオーガナイザー口座あり → `on_behalf_of=オーガナイザー`、`capture_method=manual`（即時課金しない）でセッション作成
- [正常系] PayPay 決済 → `on_behalf_of` なし、`capture_method=automatic`（即時課金）でセッション作成
- [異常系] オーガナイザーの Stripe 口座が審査未完了（card_payments: pending）→ 422 エラーで決済セッションを作らずに遮断
- [異常系] 必須フィールド（qr_config_id等）欠損 → 400 エラー

**現在のケース数:** 計 **93ケース**（pay-cheers.test.ts: 13、pay-matrix.test.ts: 80）

## 2. 決済完了処理（チア）の冪等性・二重課金防止

**テストの目的・背景:**  
リアルイベント会場では通信が不安定で、ファンが決済完了画面を何度もリロードしたり、ブラウザの「戻る」ボタンで再送信が発生することがある。同じ決済に対して2回処理が走っても、課金が1回のみ・トランザクションが1件のみになることを保証する。

**網羅している観点（境界値・状態）:**
- 同一 PaymentIntent ID で `/api/pay/complete` が2回呼ばれた場合
- カード決済（manual capture）と PayPay（即時 capture）の両方のパターン
- セッション未完了・未認証状態でのバリデーション

**主なテストケースと期待される結果:**
- [正常系] 初回呼び出し → トランザクション作成・チケット発行
- [冪等] 同じ決済ID で2回目の呼び出し → 同一トランザクションIDを返す。DB のレコードは1件のまま（二重課金なし）
- [異常系] 決済が完了していない（ステータス: 未払い）で `/complete` を呼ぶ → 400 エラー

**現在のケース数:** 計 **5ケース**（idempotency.test.ts の一部）

## 3. 入場チケット — タイプA（前売り・クレカ事前登録方式）

**テストの目的・背景:**  
タイプAは「イベント5日前にカードを確認し、当日5日前以降にオーソリ（仮押さえ）して、精算時に本確定」という多段リスク管理フロー。このアーキテクチャが絶対に崩れないことを機械的に保証する。  
また「開催まで何日あるか」で処理が分岐する（5日より前 → カード登録のみ、5日以内 → 即オーソリ）。

**網羅している観点（境界値・状態）:**
- 時間境界: 開催まで14日前・7日前・**5日前ちょうど**・4日前・1日前・当日の6パターン
- カード登録（SetupIntent）完了後のチケット発行フロー
- 5日前以降の直接オーソリフロー（PaymentIntent）
- 在庫との組み合わせ
- 予約ステータス（pending → reserved → charged）の遷移

**主なテストケースと期待される結果:**
- [正常系] 14日前に予約 → SetupIntent で「カード登録のみ」。チケットは未発行（`reservation.status=pending`）
- [正常系] 3日前に予約 → 直接 PaymentIntent オーソリ（`capture_method=manual`）。チケット発行
- [正常系] SetupIntent 完了コールバック → `reservation.status=reserved`・チケット発行
- [異常系] PaymentIntent が `requires_capture` 以外の状態で完了通知 → 400 エラー
- [異常系] 処理済み予約への二重コールバック → 409 Already processed

**現在のケース数:** 計 **約30ケース**（entrance.test.ts + entrance-complete.test.ts + entrance-matrix.test.ts の一部）

## 6. 在庫の排他制御（オーバーセル防止）

**テストの目的・背景:**  
人気イベントで「残り1枚」の状態に複数人が同時にアクセスしたとき、DBの排他ロック（`reserve_product_stock` RPC）が正しく機能し、1人だけが成功して残りは「売り切れ」になることを保証する。理論上は確認しているつもりでも、実際に並行実行して確かめたことがなかったため追加した。

**網羅している観点（境界値・状態）:**
- 在庫1枚 × 同時5人：1人成功・4人が SOLD_OUT を受け取る
- 在庫0枚（売り切れ状態）× 同時3人：全員 SOLD_OUT
- 在庫10枚 × 同時5人：全員成功（ロックが正常に解放される確認）

**主なテストケースと期待される結果:**
- [排他制御] 残1枚への同時5リクエスト → DB の `sold_count` が 1 のまま（2以上にならない）
- [異常系] 既に売り切れ → 即座に 409 SOLD_OUT（課金前に弾く）
- [正常系] 十分な在庫 → 全員が正常予約。`sold_count` が正確に件数分増える

**現在のケース数:** 計 **3ケース**（race-condition.test.ts）

## 9. 手数料計算・端数処理（金の精度保証）

**テストの目的・背景:**  
「1円の誤差も許さない」が本システムのコアポリシー。Stripe 手数料（カード 3.96%、PayPay 4.378%）とプラットフォーム手数料（10%）を引いた残額が、関係者に正しく分配されることを、あらゆる金額パターンで確認する。特にPayPayに消費税が乗る仕様（カードは非課税、PayPayは課税）は法律的な根拠があるため、それも検証対象とする。

**網羅している観点（境界値・状態）:**
- 境界値30パターン: 1円・100円・999円・1,000円・1,001円・2,999円・3,000円・3,001円・5,000円・10,000円・29,999円・30,000円・100,000円・150万円 など
- 手数料レート: カード(3.96%) × PayPay(4.378%) × 低プラットフォーム率(5%) の3パターン
- 分配率: 2者〜3者の13パターン（100:0、50:50、99:1、33.3:33.3:33.4 等）
- 実DB値検証: DB の `fee_config` テーブルの実際のレートで計算した結果が整合することを確認（設定値が変わったら自動検知）

**主なテストケースと期待される結果:**
- [恒等式] `net_amount + stripe_fee + platform_fee = gross（総額）` が全金額・全レートで成立
- [端数] `Math.floor` による切り捨て誤差は最大1円以内
- [PayPay] PayPay 手数料率 > カード手数料率（消費税分の上乗せが正確）
- [DB乖離] 実レートが nominal 値（3.96% 等）から30%以上ずれていたらテスト失敗（設定ミスの自動検知）

**現在のケース数:** 計 **580ケース**（tax.test.ts: 9、tax-matrix.test.ts: 542、tax-liverates.test.ts: 22 + 境界値×PayPay）

## 10. 決済精算・分配（Settle）

**テストの目的・背景:**  
イベント終了後、オーガナイザー・アーティスト・エージェントへの送金分配を実行する最も重要な処理。金額精度・複数パターンの分配・エラー時の処理を保証する。

**網羅している観点（境界値・状態）:**
- 分配対象: organizer / artist / agent の3役割
- 特殊ケース: Connect 口座未設定のアーティスト（オーガナイザーに帰属）
- 決済手段別: カード（manual capture）/ PayPay（自動 capture）の違い
- 二重精算防止
- 分割精算の合計一致（15,000円 + 5,000円 = 20,000円 一括精算と同額）

**主なテストケースと期待される結果:**
- [正常系] organizer 50% / artist 50% の場合、端数処理後も合計が `net_amount` と一致
- [正常系] agent 手数料 5% が gross から計算され、Stripe Transfer が作成される
- [正常系] Connect 口座未設定アーティスト → 分配がオーガナイザーに自動転換
- [冪等] 精算済みイベントの再精算 → 400 Already settled（二重送金防止）
- [正常系] PayPay（自動 capture 済み）→ キャプチャスキップして正常分配

**現在のケース数:** 計 **16ケース**（settle.test.ts）

## 11. 後半戦：イベント終了→照合→精算の一気通貫フロー

**テストの目的・背景:**  
「イベント終了 → Stripe と DB のログ全件照合 → エビデンス審査 → Settle（分配）」という後半フローを一連の流れで検証。特に「エビデンスがなければ精算できない」というロックが機械的に保証されることが重要。

**網羅している観点（境界値・状態）:**
- イベント終了ステータス遷移
- Stripe 入金金額と DB 記録の照合（一致・不一致の両パターン）
- エビデンス未提出状態でのロック
- 2週間チャージバック待機期間ロック（分配が `accrued` 状態）
- 二重精算防止

**主なテストケースと期待される結果:**
- [正常系] `ended` 状態で照合実行 → `amount_verified=true`、`reconciled_at` が記録される
- [正常系] 照合 → エビデンス提出 → Settle → `distributions` が全員 `accrued`（出金前ロック）
- [ロック] エビデンスなしで Settle → 400 No evidence submitted（出金不可）
- [異常系] Stripe 金額（1万円）≠ DB 金額（9千円）→ `amount_verified=false`・`amount_mismatch=1000` として記録

**現在のケース数:** 計 **11ケース**（settle-flow.test.ts）

## 12. エビデンス審査フロー

**網羅している観点（境界値・状態）:**
- エビデンス提出の前提条件（イベントが終了済みであること）

**主なテストケースと期待される結果:**

## 13. 照合差分検知・Payout ガード（汚染テスト）

**テストの目的・背景:**  
出金に必要な全条件を「1項目ずつ意図的に破壊」して、ガードが正しく機能するか確認する「汚染テスト」。「照合済み・金額一致・凍結なし・ホールド解放済み」という複数条件が全部揃って初めて出金できる設計を保証する。

**網羅している観点（境界値・状態）:**
- 照合差分: Stripe 入金 = DB 記録（一致）、差分あり（不一致）の10パターン
- Payout ガード条件を1つずつ汚染: `is_frozen=true`・`amount_verified=false`・`amount_mismatch≠0`・`reconciled_at=null` 等8パターン
- 出金申請金額バリデーション（振込手数料以下はNG等）6パターン

**主なテストケースと期待される結果:**
- [異常系] `is_frozen=true`（チャージバック中）→ 出金残高ゼロ
- [異常系] 照合未了（`reconciled_at=null`）→ 出金対象に含まれない
- [異常系] 振込手数料（500円）以下の金額で出金申請 → 400 エラー
- [正常系] 全条件クリア → 出金成功

**現在のケース数:** 計 **32ケース**（post-pay-matrix.test.ts）

## 15. 返金処理（5パターン）

**テストの目的・背景:**  
「いつ・どのポリシーで返金するか」によって処理が全く異なる。プラットフォームの損益も変わるため、5パターン全ての分岐が正確に動くことを保証する。

**網羅している観点（境界値・状態）:**

| パターン | 状態 | ポリシー | 処理 |
|---------|------|---------|------|
| ① | オーソリ中（未課金） | — | PI キャンセル（資金移動ゼロ） |
| ② | 課金済み・精算前 | 全額保証（FULL_PENALTY） | 返金 + プラットフォーム手数料を請求 |
| ③ | 課金済み・精算前 | 同情的（COMPASSIONATE） | 返金 + Stripe 手数料のみ請求（プラットフォーム無傷） |
| ④ | 課金済み・精算後 | 全額保証 | 返金 + 全送金逆転 + プラットフォーム手数料請求 |
| ⑤ | 課金済み・精算後 | 同情的 | 返金 + 全送金逆転 + Stripe 手数料のみ請求 |

- バリデーション: 入力パラメータ欠損・無効な返金タイプ・管理者権限チェック
- PI 状態マトリクス（requires_capture / succeeded / canceled）

**主なテストケースと期待される結果:**
- [正常系 ①] オーソリ中の PI → キャンセル、`transaction.status=cancelled`

## 17. イベントライフサイクル管理

**テストの目的・背景:**  
イベントには「下書き → 審査中 → 公開 → 終了 → 精算済み」という正規の流れがあり、それ以外の順番で操作しようとした場合は弾かれる。例えば「精算済みイベントをキャンセルしようとする」「承認前のイベントを終了しようとする」といった操作が全て 400 エラーになることを保証する。

**網羅している観点（境界値・状態）:**

| 操作 | 許可される状態 | 拒否される状態（400） |
|-----|------------|---------------|
| 審査依頼 | draft のみ | review_requested / published / ended / settled / cancelled |
| 承認 | review_requested のみ | それ以外全て |
| 終了 | published / ongoing のみ | draft / ended / settled / cancelled |
| キャンセル申請 | draft〜ongoing | ended / settled / cancelled |
| 精算 | 証拠あり・未精算 | settled（二重精算防止） |

- イベント作成のバリデーション（タイトル必須・オーガナイザー限定・担当エージェント必要）
- 中止申請〜エージェント承認/却下のフロー
- セルフ承認防止（オーガナイザー兼エージェントの場合は管理者のみ承認可）

**主なテストケースと期待される結果:**
- [異常系] published イベントに review_requested → 400「ドラフト状態のみ審査依頼可」
- [異常系] settled イベントをキャンセル → 400
- [正常系] draft イベントを即キャンセル → `lifecycle_status=cancelled`（エージェント承認不要）
- [正常系] published イベントのキャンセル申請 → `cancellation_requested`（エージェント承認待ち）
- [正常系] エージェントが却下 → `published` に戻る

**現在のケース数:** 計 **41ケース**（events-lifecycle.test.ts: 21、lifecycle-matrix.test.ts: 20）

## ■ 「このテストで守れないこと」（既知の限界）

1. **価格上限のバックエンド検証** — スタンダード3,000円・メッセージ5,000円・入場30,000円の上限はフロントエンドのみで制御。APIに直接リクエストを送れば上限超えが可能。将来のバックエンド検証実装後にテスト追加予定。

2. **PayPay のリアル決済** — Stripe テストモードに PayPay のサンドボックスがないため、パラメータ検証のみ。実際のPayPay課金フローは手動確認が必要。

3. **ウォレット（Apple Wallet）** — 外部サービス依存のため自動テスト不可。

4. **メッセージ・フォロー・プロフィール更新** — 低リスクと判断し未テスト。問題が発生した際に追加。
